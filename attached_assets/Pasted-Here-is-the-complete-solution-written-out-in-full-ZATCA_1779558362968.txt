## Bug #1 ŌĆö Wrong Elliptic Curve (crypto.ts : line 209)

**Severity: CRITICAL**

ZATCA mandates the `secp256r1` curve (also called `prime256v1` / NIST P-256) for all EGS private keys. Your code currently uses `secp256k1`, which is Bitcoin's curve. ZATCA's CA will reject every CSR generated with this key.

**Current code (line 209):**
```ts
execSync(`openssl ecparam -name secp256k1 -genkey -noout -out "${keyPath}"`, { stdio: "pipe" });
```

**Fixed code:**
```ts
execSync(`openssl ecparam -name prime256v1 -genkey -noout -out "${keyPath}"`, { stdio: "pipe" });
```

That is the entire change ŌĆö one word. However, the consequence is significant: every private key and CSR generated before this fix is permanently invalid. You must delete all stored `privateKey`, `csr`, `complianceCsid`, and `productionCsid` values for every registered account, and re-run the full onboarding flow from scratch for each one.

After regenerating, verify the curve with:
```bash
openssl req -in csr.pem -text -noout | grep "Public Key Algorithm"
# Expected: Public Key Algorithm: id-ecPublicKey
#           ASN1 OID: prime256v1
```

## Bug #2 ŌĆö Incorrect Invoice Type Codes (xml-generator.ts : lines 148ŌĆō153)

**Severity: CRITICAL**

ZATCA's three invoice type codes are: `388` = Tax Invoice, `381` = Credit Note, `383` = Debit Note. The function `generateInvoiceTypeCode()` has a broken fallthrough condition on line 151 that maps `invoiceSubType === '02'` to `383` for standard invoices, which silently mislabels some invoices and causes `KSA-2 / BT-3` validation failures during compliance checks.

**Current code (broken):**
```ts
export function generateInvoiceTypeCode(
  invoiceType: "standard" | "simplified",
  invoiceSubType: "01" | "02",
  documentType?: "invoice" | "credit_note" | "debit_note"
): string {
  if (documentType === "credit_note") return "381";
  if (documentType === "debit_note")  return "383";
  if (invoiceSubType === "02")
    return invoiceType === "standard" ? "383" : "381";  // ŌåÉ BUG
  return "388";
}
```

The third condition is wrong. `invoiceSubType` is a document-level flag that does not change the type code. The type code is determined solely by `documentType`.

**Fixed code:**
```ts
export function generateInvoiceTypeCode(
  invoiceType: "standard" | "simplified",
  invoiceSubType: "01" | "02",
  documentType?: "invoice" | "credit_note" | "debit_note"
): string {
  if (documentType === "credit_note") return "381";
  if (documentType === "debit_note")  return "383";
  return "388"; // Standard tax invoice ŌĆö both standard and simplified
}
```

## Bug #3 ŌĆö Hash Recomputed After SDK Signing on Retry (service.ts : line 425)

**Severity: CRITICAL**

When the ZATCA Java SDK signs an invoice, it computes the canonical hash using its own C14N engine. In the main `processInvoiceForZatca()` path, the code correctly avoids recomputing the hash after SDK signing (the `usedSdk` guard at lines 256ŌĆō261). However, the `retryPendingInvoices()` function at line 425 always recomputes the hash from the stored XML using your local canonicalizer, regardless of whether the SDK was originally used. If the local canonicalizer produces slightly different canonical bytes than the SDK did, the submitted hash will not match the invoice content and ZATCA will reject it.

**Current code (broken):**
```ts
for (const invoice of pendingInvoices) {
  if (!invoice.signedXml) continue;
  processed++;

  // Always recomputes ŌĆö wrong when SDK was used originally
  const recomputedHash = generateInvoiceHash(invoice.signedXml);

  const result = await submitInvoiceToZatca(
    config,
    invoice.signedXml,
    recomputedHash,
    invoice.uuid,
    invoice.invoiceType as "standard" | "simplified"
  );
}
```

**Fixed code:**
```ts
for (const invoice of pendingInvoices) {
  if (!invoice.signedXml) continue;
  processed++;

  // Use the hash that was stored at original signing time.
  // Fall back to recomputing only for legacy rows that predate this fix.
  const hashToSubmit = invoice.invoiceHash
    || generateInvoiceHash(invoice.signedXml);

  const result = await submitInvoiceToZatca(
    config,
    invoice.signedXml,
    hashToSubmit,
    invoice.uuid,
    invoice.invoiceType as "standard" | "simplified"
  );
}
```

For this fix to work, `invoiceHash` must be stored in the database at submission time. Check your `storage.createInvoiceZatcaStatus()` call ŌĆö the field is already being passed in the main path at line 344. Confirm your database schema and query in `getInvoiceZatcaStatuses()` both include and return the `invoiceHash` column.

## Issue #4 ŌĆö Duplicate `<cac:TaxTotal>` Element (xml-generator.ts : ~line 439)

**Severity: HIGH**

The generated invoice XML contains two consecutive `<cac:TaxTotal>` elements at the invoice level. ZATCA's UBL 2.1 schema allows exactly one. The first block has only a `TaxAmount` child; the second has a `TaxAmount` plus the required `TaxSubtotal`. The first one must be removed.

**Current output (wrong):**
```xml
<!-- Remove this entire block -->
<cac:TaxTotal>
  <cbc:TaxAmount currencyID="SAR">15.00</cbc:TaxAmount>
</cac:TaxTotal>

<!-- Keep only this one -->
<cac:TaxTotal>
  <cbc:TaxAmount currencyID="SAR">15.00</cbc:TaxAmount>
  <cac:TaxSubtotal>
    <cbc:TaxableAmount currencyID="SAR">100.00</cbc:TaxableAmount>
    <cbc:TaxAmount currencyID="SAR">15.00</cbc:TaxAmount>
    <cac:TaxCategory>
      <cbc:ID schemeAgencyID="6" schemeID="UN/ECE 5305">S</cbc:ID>
      <cbc:Percent>15</cbc:Percent>
      <cac:TaxScheme>
        <cbc:ID schemeAgencyID="6" schemeID="UN/ECE 5153">VAT</cbc:ID>
      </cac:TaxScheme>
    </cac:TaxCategory>
  </cac:TaxSubtotal>
</cac:TaxTotal>
```

In `buildInvoiceBodyParts()`, find the `supplierAndAfter` template string. Delete the first `<cac:TaxTotal>` block (the three-line one without `TaxSubtotal`). The final invoice must have only one `<cac:TaxTotal>` element, which is the one containing `<cac:TaxSubtotal>`.

## Issue #5 ŌĆö `generateInvoiceTypeCodeName()` Ignores `documentType` (xml-generator.ts : lines 155ŌĆō157)

**Severity: HIGH**

The `name` attribute on `<cbc:InvoiceTypeCode>` is a 7-character bitmask. Position 1 encodes the document sub-type: `0` = tax invoice, `2` = credit note, `3` = debit note. The current function only looks at `invoiceType` (standard/simplified) and ignores `documentType`, so credit notes from standard invoice flows produce `name="0100000"` instead of the required `"0200000"`.

**Current code (wrong):**
```ts
export function generateInvoiceTypeCodeName(
  invoiceType: "standard" | "simplified"
): string {
  return invoiceType === "standard" ? "0100000" : "0200000";
}
```

**Fixed code:**
```ts
export function generateInvoiceTypeCodeName(
  invoiceType: "standard" | "simplified",
  documentType?: "invoice" | "credit_note" | "debit_note"
): string {
  if (documentType === "credit_note") return "0200000";
  if (documentType === "debit_note")  return "0300000";
  return invoiceType === "standard" ? "0100000" : "0200000";
}
```

Also update the call site inside `buildInvoiceBodyParts()`:

**Before:**
```ts
const invoiceTypeCodeName = generateInvoiceTypeCodeName(invoiceType);
```

**After:**
```ts
const invoiceTypeCodeName = generateInvoiceTypeCodeName(invoiceType, documentType);
```

## Issue #6 ŌĆö Invoice Hash Encoding: 88-char vs 44-char (xml-generator.ts : lines 216ŌĆō218)

**Severity: HIGH**

There are two conventions used in the field for the `DigestValue` in ZATCA invoices:

- **Convention A (44 chars):** `base64( raw_32_SHA256_bytes )` ŌĆö used by ZATCA's own Java SDK.
- **Convention B (88 chars):** `base64( lowercase_hex_string_of_SHA256 )` ŌĆö used by some third-party implementations.

Your implementation uses Convention B. If ZATCA's validator rejects your invoices with a "hash does not match" error, switch to Convention A.

**Current code ŌĆö Convention B (88 chars):**
```ts
export function generateInvoiceHash(xmlContent: string): string {
  const hex = crypto
    .createHash("sha256")
    .update(canonicalizeInvoiceXml(xmlContent), "utf8")
    .digest("hex");                              // 64-char hex string
  return Buffer.from(hex, "utf8").toString("base64"); // base64 of that hex ŌåÆ 88 chars
}
```

**Alternative ŌĆö Convention A (44 chars), try this if ZATCA rejects:**
```ts
export function generateInvoiceHash(xmlContent: string): string {
  return crypto
    .createHash("sha256")
    .update(canonicalizeInvoiceXml(xmlContent), "utf8")
    .digest("base64"); // base64 of raw 32-byte digest ŌåÆ 44 chars
}
```

Test both in ZATCA's simulation environment. Submit one invoice with each. The one that returns `CLEARED` or `REPORTED` without a hash mismatch error is the correct form for your environment. If you switch to Convention A, ensure `generateInvoiceHashHex()` is still consistent and that QR Tag 6 still passes raw bytes (not the hex string bytes) into the TLV buffer.

## Issue #7 ŌĆö Legacy Regex Canonicalizer Still Reachable (crypto.ts : lines 582ŌĆō601)

**Severity: MEDIUM**

`crypto.ts` contains a `canonicalizeXML()` function that uses regular expressions to normalize XML. This is not a compliant C14N implementation ŌĆö it does not handle namespace inheritance, attribute ordering, or character encoding edge cases required by the XML Canonicalization 1.0 specification. `xml-generator.ts` correctly uses `@xmldom/xmldom` plus `xml-crypto`'s `C14nCanonicalization` class, which is compliant. The problem is that `signCanonicalXML()` in `crypto.ts` (line 279) still calls the regex version and may be reachable from legacy code paths.

**Step 1 ŌĆö audit all callers:**

## Issue #8 ŌĆö `generateInvoiceCounter()` Uses `Date.now()` (crypto.ts : line 627)

**Severity: MEDIUM**

The ICV (Invoice Counter Value) must be a sequential integer starting at 1 and incrementing by exactly 1 per invoice per EGS device. `Date.now()` returns a Unix millisecond timestamp (e.g. `1716480000000`), which is not sequential in ZATCA's sense and will trigger `KSA-16` validation errors.

**Current code (wrong for ICV):**
```ts
export function generateInvoiceCounter(): number {
  return Date.now();
}
```

The correct counter is already available in `service.ts` at line 147 from `storage.incrementInvoiceCounter()`. Confirm that `baseInvoiceData.invoiceCounter` is always set to that value and that no code path calls `generateInvoiceCounter()` for real invoice submissions.

**Mark the function deprecated:**
```ts
/**
 * @deprecated Do NOT use for ICV (Invoice Counter Value).
 * ICV must be a sequential integer: 1, 2, 3, ...
 * Use the counter returned by storage.incrementInvoiceCounter() instead.
 */
export function generateInvoiceCounter(): number {
  return Date.now();
}
```

In `service.ts`, the counter is already correct ŌĆö verify it is always used:

## Issue #9 ŌĆö No Proactive CSID Age Check (service.ts)

**Severity: LOW**

Compliance CSIDs expire after 1 hour. The current code detects expiry only by catching 401 responses after the fact. A better approach warns the user before submissions when the CSID is near expiry.

**Add this at the start of the compliance check function, before the invoice loop:**
```ts
// Store this when saving a new Compliance CSID:
// await storage.updateZatcaSettings(restaurantId, { csidIssuedAt: new Date() });

const csidIssuedAt: Date | null = settings.csidIssuedAt || null;
const csidAgeMinutes = csidIssuedAt
  ? (Date.now() - csidIssuedAt.getTime()) / 60000
  : null;

if (csidAgeMinutes !== null && csidAgeMinutes > 55) {
  return {
    success: false,
    results: [{
      invoiceType: "all",
      passed: false,
      errors: [{
        code: "CSID_NEAR_EXPIRY",
        message:
          `Compliance CSID is ${Math.round(csidAgeMinutes)} minutes old and likely expired. ` +
          "Please generate a new OTP and request a new Compliance CSID before retrying."
      }]
    }]
  };
}
```
