The ZATCA Phase 2 onboarding and invoice submission process must follow these exact steps in order. Review the current implementation and fix any step that is out of order, missing, or incorrect:
Step 1: User fills in VAT number, organization name, branch name, common name, invoice type (1000/0100/1100), address fields, CR number, solution name, and device serial number.
Step 2: System generates a prime256v1 EC private key and a CSR containing those details. Store both in the database. Private key never leaves the server.
Step 3: User manually gets a one-time OTP from fatoora.zatca.gov.sa and pastes it into the app.
Step 4: System submits the CSR and OTP to ZATCA /compliance endpoint. Store the returned binarySecurityToken as complianceCsid, secret as complianceCsidSecret, and requestID as complianceRequestId. Also store csidIssuedAt = new Date().
Step 5: System automatically runs 6 compliance test invoices using the Compliance CSID. The 6 types are: standard invoice, standard debit note, standard credit note, simplified invoice, simplified debit note, simplified credit note. All must return no errors. The InvoiceTypeCode name attribute must be 010000000 for standard and 020000000 for simplified (9 characters). Credit and debit notes must include <cbc:Note> with the adjustment reason at the invoice header level immediately after <cbc:InvoiceTypeCode>.
Step 6: Only after all 6 compliance checks pass, system automatically calls /production/csids using the complianceRequestId. Store the returned token as productionCsid, secret as productionCsidSecret. Set isEnabled = true and onboardingStatus = "production_ready".
Step 7: Every real invoice must: generate UUID, get sequential counter from database, get previous invoice hash from database, build and sign XML, compute hash, generate Phase 2 QR (9 TLV tags), submit standard invoices to /invoices/clearance/single with header Clearance-Status: 1, and simplified invoices to /invoices/reporting/single with header Clearance-Status: 0. For cleared standard invoices, store the XML that ZATCA returns — not the locally signed one.
Step 8: On retry, always use the invoiceHash stored in the database at signing time. Never recompute the hash from the stored XML on retry.
Do not change any non-ZATCA features. Only fix the ZATCA integration flow.
